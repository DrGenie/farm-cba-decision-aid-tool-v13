Quick start

The tool is designed to support early conversations about trial results, not to replace a full economic evaluation. It aims to keep users engaged even when data are imperfect.

  1. Download the template. The Data & settings tab provides a blank template (TSV or Excel) with all required column headers, including a flag for control rows.
  2. Enter one row per plot or replicate. Each row should record the treatment name, whether it is a control, and the core cost and outcome fields (for example yield and total cost per hectare).
  3. Upload and check. Use “Check file” to see whether required columns, control rows, and treatment rows have been recognised. The tool explains what is missing if something is wrong.
  4. Run the analysis. Once the file passes the structural checks, click “Upload and use in analysis” and then use the “Run analysis” button in the Results tab if you want to re-run the calculations after changing assumptions.
  5. Export or use AI. The Exports & AI brief tab can generate a Word document and an AI-ready prompt that can be pasted into Copilot or ChatGPT for a short narrative summary.

Better structured data gives better analysis. However, the tool can still provide basic, clearly labelled indicative results when some optional fields are missing or when only a subset of plots has complete data.

Minimum data required

The decision aid enforces only a small set of minimum requirements. This is intentional: it reduces the risk that users stop using the tool because their dataset is not perfect.

  • At least one control row. The is_control column must flag at least one row as control (values such as 1, true, yes, or control are accepted).
  • At least one treatment row. There must be at least one non-control row to compare against the control.
  • Core cost and outcome fields present. The template includes columns for yield per hectare and total cost per hectare. At least some rows must contain usable numbers in these fields.

Optional columns (for example variable cost, fixed cost, capital cost, or other benefit) may be left blank. The tool does not require users to delete extra columns, and it does not block uploads because of missing optional values.

If some optional fields are missing or partly populated, the tool falls back to a “basic analysis”. It reports treatment means and simple differences relative to the control and labels these outputs as indicative. This helps keep users engaged rather than leaving them with an empty results screen.

How data are read and validated

Data can be uploaded as TSV, CSV, or Excel files. The tool reads the first row as the header and expects one row per plot or replicate. The validation step performs three main checks before the analysis runs.

  1. Header names. Column headers are normalised (lowercase, spaces to underscores) and compared with the required names in the template. If required columns are missing, the tool reports “Missing required column(s)” and lists them explicitly.
  2. Control and treatments. The is_control column is used to identify control rows. If no control row is found, the tool reports “No control identified”. If all rows are flagged as control, it reports “No treatment rows detected”.
  3. Numeric fields. Yield and total cost columns are checked for usable numeric values. If all entries in a core field are non-numeric or empty, the tool reports that these fields do not contain usable values and asks the user to review formatting.

The “Check file” button runs these validations and shows a list of errors and warnings without altering the current dataset. Only when the user clicks “Upload and use in analysis” and the file passes the core checks does the tool replace the current data.

When a file does not meet the minimum requirements, the tool explains what is wrong in plain language (for example: missing required columns, no control identified, or no treatment rows). This is designed to prevent the common reaction “My data will not load, and I do not know how to fix it”.

How replicates are aggregated

Most field trials and demonstrations have multiple plots or replicates for each treatment. The decision aid keeps the data structure simple:

  • Each row in the dataset represents one plot or replicate.
  • All rows with the same treatment name are grouped together.
  • For each treatment, the tool calculates average yield, average total cost, and any other available average values per hectare.

At the analysis stage, every table and chart is based on these aggregated treatment averages. The control treatment is also summarised in the same way, using all rows flagged as control. This is why the Results tab explicitly notes that treatments are compared using averages over replicates.

If some plots have missing values in optional columns, those values are simply excluded from the average for that column. The presence of missing data in optional fields does not stop the analysis from running.

How indicators are calculated

The tool calculates a small set of standard economic indicators for each treatment: net present value (NPV), benefit-cost ratio (BCR), payback period, and net annual return per hectare. All calculations are done per hectare using the averages described above and the scenario assumptions set by the user.

  • Net annual return per hectare. The tool multiplies average yield per hectare by the user-entered grain price to obtain revenue per hectare, then subtracts total cost per hectare to obtain net annual return per hectare.
  • Net present value (NPV). If enough information is available, the tool applies the chosen time horizon and discount rate to convert the stream of net annual returns (and any additional benefits or capital costs) into a net present value per hectare.
  • Benefit-cost ratio (BCR). The benefit-cost ratio is calculated as the present value of benefits divided by the present value of costs. Where only partial information is available, this is left blank rather than forcing a misleading number.
  • Payback period. When a capital cost per hectare is provided, the tool calculates an approximate payback period by dividing capital outlay by net annual return per hectare. If net returns are non-positive, payback is not reported.

All treatments are then compared with the selected control treatment. The comparison table in the Results tab reports, for each indicator, the level for the control, the level for each treatment, and the difference (treatment minus control).

When information required for full discounted cost-benefit analysis is missing, the tool does not fabricate detail. Instead, it focuses on reporting simple treatment means and differences, and labels the output as indicative.

Handling incomplete or messy data

Trial datasets are often imperfect. Plots may be missing, some variables may be recorded only in a subset of sites, and column names may differ between projects. The decision aid is built to respond to this reality rather than requiring a perfectly cleaned file.

  • Missing optional values. Optional columns can be entirely empty or partially filled. Calculations based on those fields use only rows with usable numbers and leave indicators blank where a meaningful figure cannot be derived.
  • Extra columns. Additional columns beyond those in the template are ignored by the analysis. Users do not need to delete them.
  • Basic analysis fallback. If full discounted analysis is not possible (for example, because capital costs or additional benefits are not provided), the Results tab displays a basic analysis: treatment means, simple differences relative to the control, and a clear note that results are indicative.

This design aims to keep users engaged even when they have limited time or experience with data cleaning. It also helps farming systems groups use the tool across a variety of trial datasets without needing to standardise every detail in advance.

Exports and AI-assisted summaries

The Exports & AI brief tab has been designed to make it easy to move from on-screen results to documents and short written summaries.

  • Word report exports. The “Results summary” export creates a Word document containing the main comparison table, replicate summary, and a short technical description of the scenario assumptions. The “Technical appendix” export provides a concise description of how the indicators were calculated.
  • AI-ready prompt. The AI prompt box contains a structured description of the scenario and results (including a simple table in tab-separated format). The “Copy prompt” button copies this text to the clipboard.
  • Opening Copilot or ChatGPT. The “Open in Copilot” and “Open in ChatGPT” buttons open a new browser window on the relevant site and copy the AI prompt so users can paste it into the chat box. The prompt is written to encourage short, plain-language summaries that are suitable for growers, advisors, or project reports.

The AI integration is deliberately simple: the tool prepares a clear, data-linked prompt and leaves the wording of the final narrative to the AI system. This helps ensure that summaries are grounded in the numbers produced by the decision aid while still being quick to adapt for different audiences.

Troubleshooting and good practice

A few patterns help users get the most value from the decision aid while keeping effort manageable.

  • Start from the template. When in doubt, download a fresh template, paste or merge your existing data into it, and then upload this file. This keeps header names and structure consistent.
  • Use descriptive treatment names. Treatment names should be clear enough that users can recognise them easily in the Results tab and exports.
  • Document assumptions. When using outputs in reports or extension materials, record the key assumptions: grain price, time horizon, discount rate, and what is included in costs. The Word exports and AI prompt both include these details to help.
  • Review indicative results carefully. When the tool labels results as indicative, take time to check which data are missing. In some cases a small amount of additional data cleaning or re-entry can turn an indicative comparison into a full cost-benefit analysis.

The most common adoption risk identified during user testing is early disengagement at the data-loading stage. Clear onboarding, realistic minimum requirements, and direct feedback about what needs to be fixed are therefore central design features of this tool.
